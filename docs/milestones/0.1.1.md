# 0.1.1 Breakdown

This milestone is intentionally narrow.

It should implement only the first workflow from [../workflows/01-single-asset-no-inputs.md](../workflows/01-single-asset-no-inputs.md), while setting up the core Rust, Python, and frontend structure that later workflows will extend.

## Goal

Prove the Barca core loop on the smallest honest slice:

- discover one decorated asset
- persist its definition
- materialize it via a Rust-managed Python worker
- write its artifact to `.barcafiles`
- show it in a basic Datastar web UI
- reuse cache when its definition has not changed

## What 0.1.1 should implement

### Python SDK foundation

Implement the minimum Python package needed for the first workflow:

- `@asset`
- metadata attachment on decorated functions
- inspection command for no-input assets
- worker command for executing one no-input asset
- JSON serialization for simple JSON-serializable outputs

The asset shape to support is exactly:

```python
from barca import asset


@asset()
def a() -> str:
    return "banana"
```

### Rust orchestrator foundation

Implement the minimum Rust application needed to orchestrate that workflow:

- `tokio` application entrypoint
- HTTP server
- metadata store abstraction
- indexing service
- execution service
- artifact publication logic
- basic live status propagation for the web UI

Rust should be authoritative for:

- indexing orchestration
- definition and materialization persistence
- cache lookup
- worker spawning
- current run state

### Datastar frontend foundation

Implement a small operator-facing UI:

- homepage listing indexed assets
- asset detail page
- button to trigger materialization
- visible current/latest materialization state
- basic live updating status while a run is in progress

This should be enough to prove:

- Barca can render autodiscovered assets
- the Rust app can stream state to the browser
- the UI model matches the orchestration model

## Recommended repo shape

For 0.1.1, the repo should end up with three clear surfaces:

### Rust app

Suggested responsibilities:

- binary entrypoint
- API routes
- Datastar fragment routes
- metadata access
- index orchestration
- execution orchestration

### Python package

Suggested responsibilities:

- decorators
- inspection
- worker entrypoint
- JSON serialization

### Web UI templates/fragments

Suggested responsibilities:

- asset list
- asset detail
- run status fragment

## Concrete vertical slice

The end-to-end path for 0.1.1 should be:

1. User defines `@asset() def a() -> str: return "banana"`.
2. Rust indexing flow launches Python inspection.
3. Python returns metadata for `a`.
4. Rust computes/stores:
   - continuity key
   - `definition_hash`
   - asset definition record
5. Web UI shows asset `a`.
6. User triggers “materialize”.
7. Rust checks for matching successful materialization.
8. If none exists, Rust launches Python worker via `uv run`.
9. Python imports the real module, calls `a()`, serializes `"banana"` to JSON.
10. Rust publishes:
    - `.barcafiles/<slug>/<definition-hash>/code.txt`
    - `.barcafiles/<slug>/<definition-hash>/metadata.json`
    - `.barcafiles/<slug>/<definition-hash>/value.json`
11. Rust records the materialization.
12. UI updates to show success.
13. Re-running with unchanged definition reuses the existing materialization.

## Rust/Python boundary for 0.1.1

Keep the boundary minimal.

### Rust -> Python inspection

Pass:

- configured source roots
- target package/module roots

Return:

- discovered assets
- module path
- file path
- function name
- function source
- decorator metadata
- return annotation

### Rust -> Python worker

Pass:

- module path
- function name
- expected `definition_hash`
- output staging directory
- serializer kind

Return:

- success/failure
- artifact manifest
- output metadata
- error payload if any

## Metadata scope for 0.1.1

Only implement the minimum tables/records needed for the first workflow:

- `assets`
- `asset_definitions`
- `materializations`

The metadata model should already be append-only, even though the workflow is simple.

## UI scope for 0.1.1

Do not overbuild the UI.

Needed:

- list assets
- open one asset
- trigger materialization
- see latest status

Not needed yet:

- graph view
- lineage explorer
- partitions UI
- sensors/effects UI
- history-heavy views

## Explicitly out of scope

0.1.1 should not try to partially implement future workflows.

Out of scope:

- asset dependencies
- `load_inputs(...)`
- partitions
- sensors
- effects
- schedules and reconciler loop
- retries
- timeouts
- cancellation
- ad hoc params
- notebook-specific helpers
- parquet/pickle support

Those belong in later milestones.

## Why this is the right cut

This slice is small enough to finish, but still real enough to validate the architecture.

If 0.1.1 works, we will have proven:

- the Rust/Python boundary is sound
- the metadata model is viable
- the artifact layout is viable
- the Datastar UI can represent live orchestration state

That is enough foundation to add the second workflow next without rewriting the whole system.

## Acceptance criteria

- A no-input asset can be discovered from Python decorator metadata.
- The asset definition can be indexed and stored by Rust.
- The asset can be materialized through a Python worker launched by Rust.
- The output is written to the expected `.barcafiles` structure.
- Re-running the same definition reuses cache.
- The Datastar web UI lists the asset and shows its latest materialization state.
- The UI can trigger a materialization and reflect the result.
