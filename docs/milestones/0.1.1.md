# 0.1.1 Breakdown

This milestone is intentionally narrow.

It implements only the first workflow from [../workflows/01-single-asset-no-inputs.md](../workflows/01-single-asset-no-inputs.md), while setting up the core Python package structure that later workflows extend.

## Goal

Prove the Barca core loop on the smallest honest slice:

- discover one decorated asset
- persist its definition
- materialize it via direct function call
- write its artifact to `.barcafiles`
- show it via CLI
- reuse cache when its definition has not changed

## What 0.1.1 implements

### Python SDK foundation

The minimum Python package needed for the first workflow:

- `@asset`
- metadata attachment on decorated functions
- inspection helper for no-input assets
- JSON serialization for simple JSON-serializable outputs

The asset shape to support:

```python
from barca import asset

@asset()
def a() -> str:
    return "banana"
```

### Core library foundation

The minimum engine needed to orchestrate:

- module discovery and import via `importlib`
- metadata store (SQLite)
- indexing service
- execution via direct function call
- artifact publication
- content-addressed caching

### CLI foundation

Basic operator interface:

- `barca reindex` — discover assets
- `barca assets list` — list indexed assets
- `barca assets show <id>` — asset detail
- `barca assets refresh <id>` — trigger materialization
- `barca reset` — clean generated files

## Concrete vertical slice

The end-to-end path for 0.1.1:

1. User defines `@asset() def a() -> str: return "banana"`.
2. `reindex()` imports the module, discovers `a`, computes `definition_hash`.
3. Definition is stored in SQLite.
4. User runs `barca assets refresh 1`.
5. Engine checks for matching successful materialization (cache).
6. If none exists, imports module, calls `a()`, serializes result to JSON.
7. Artifact written to `.barcafiles/<slug>/<definition-hash>/value.json`.
8. Materialization recorded in DB.
9. Re-running with unchanged definition reuses the existing materialization.

## Metadata scope for 0.1.1

Only the minimum tables:

- `assets`
- `asset_definitions`
- `materializations`

The metadata model is already append-only, even though the workflow is simple.

## Explicitly out of scope

0.1.1 does not attempt to partially implement future workflows:

- asset dependencies
- `load_inputs(...)`
- partitions
- sensors
- effects
- schedules and reconciler loop
- retries, timeouts, cancellation
- ad hoc params
- notebook-specific helpers
- parquet/pickle support
- HTTP server

## Acceptance criteria

- A no-input asset can be discovered from Python decorator metadata.
- The asset definition can be indexed and stored.
- The asset can be materialized through direct function call.
- The output is written to the expected `.barcafiles` structure.
- Re-running the same definition reuses cache.
- The CLI lists the asset and shows its materialization state.
